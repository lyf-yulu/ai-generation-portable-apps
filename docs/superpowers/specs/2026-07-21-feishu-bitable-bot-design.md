# 飞书多维表格任务与机器人入口设计规格

日期：2026-07-21

## 1. 背景

`feishu-generation-agent/` 已经具备一条独立于现有 Portal 的本地 LangGraph 工作流：读取飞书文档及图片，使用 Claude Vision 和 DeepSeek 理解、拆解需求，经人工审批后调用 Chiyun 图生图或火山方舟 Seedance 图生视频，并持久化供应商任务与恢复状态。

当前入口仍以“在本地页面粘贴单个飞书文档链接”为主，交付方式是新建飞书文档。新的业务入口改为一个固定的飞书多维表格：每行代表一个待处理需求，`需求来源` 指向真实需求文档，领取者写入 `执行人`，生成的图片和视频直接写入该行的 `结果`附件列。同时，除本地页面外，公司成员还应能通过 `@` 飞书应用机器人完成扫描、领取、审批和查看结果。

本次是在现有 Agent 上增加“多维表格任务模式”和“飞书机器人入口”，不另建一套生成工作流，也不接入现有 Seedance/Nano Banana 子应用。

## 2. 目标

1. 本地页面可以扫描配置好的多维表格，列出可领取的任务，并由本地操作者选择一行开始处理。
2. 飞书应用机器人通过长连接接收群聊 `@机器人` 或单聊消息，使其他成员无需访问本地网页也能使用同一功能。
3. 所有有权使用机器人的成员都可以看到并领取全部可处理记录，不按原 `执行人` 限制可见性。
4. 领取成功后把实际领取人写入 `执行人`，并用本地数据库原子锁防止两人同时领取同一行。
5. 复用现有 Claude Vision、DeepSeek、LangGraph 审批中断、Chiyun、Seedance、产物校验和付费调用幂等逻辑。
6. 飞书交互卡片展示完整计划，允许选择任务、批准、取消，以及提交反馈后重新规划。
7. 所有付费生成调用必须发生在有效人工审批之后。
8. 图片和视频直接上传到原记录的 `结果`附件列，不在多维表格模式下新建交付文档。
9. 进程重启后可以恢复处理中、待审批、生成中和回写中的任务，且不会重复付费提交。
10. 首先在用户 Mac 本地运行、测试通过；设计保留将来迁移到常驻主机或公网服务的空间。

## 3. 非目标

本次不实现：

- 定时轮询并自动领取表格任务；扫描仍由本地按钮或机器人命令触发。
- 一个进程同时管理多个多维表格；首版只配置一个固定表格和视图。
- 在机器人卡片内直接逐字段编辑提示词、参数或替换参考图。机器人首版通过“反馈并重新规划”修改方案；本地审批页保留完整编辑能力。
- 为多维表格任务新建飞书交付文档、输出文件夹或交付文档协作者。
- 绕过人工审批的全自动付费执行。
- 公网回调地址、内网穿透、局域网开放或账号系统。机器人使用飞书长连接，不要求公网入站。
- 重写现有生成适配器，或使用火山方舟 AK/SK 代替当前 Seedance Ark Bearer Key 调用链。
- 文生图、文生视频、剪辑、配音等现有 Agent 范围之外的任务类型。

现有“粘贴文档链接 → 新建飞书交付文档”流程可以作为兼容入口保留，但不再是多维表格任务的主交付路径。

## 4. 已确认决策

| 项目 | 决策 |
|---|---|
| 核心结构 | 一个 `BitableTaskService`，本地页面与飞书机器人是两个薄入口 |
| 本地入口 | 用户点击“扫描任务”，从可处理记录中选择一行 |
| 机器人入口 | 飞书应用机器人长连接；支持单聊和群聊 `@机器人` |
| 可见性 | 所有机器人使用者可见并可领取全部可处理记录 |
| 领取 | 本地 SQLite 原子占位成功后，将实际领取人写入 `执行人` |
| 本地领取身份 | 使用配置项 `LARK_LOCAL_OPERATOR_OPEN_ID`；未配置时本地页面只可扫描，不可领取 |
| 审批 | 本地页面继续支持完整编辑；飞书卡片支持任务选择、批准、取消、反馈重规划 |
| 状态字段 | 缺失时自动创建 `状态`单选和 `错误信息`多行文本字段 |
| 结果交付 | 图片、视频直接写入现有 `结果`附件字段，允许多个附件 |
| 表格模式交付文档 | 不创建新文档，不需要输出文件夹或固定文档协作者 |
| 付费保护 | 人工审批门禁 + 一次性审批版本 + 供应商提交意图幂等 |
| 恢复 | 供应商已有官方任务 ID 时只轮询，不重新提交 |
| 部署 | 首版应用仍绑定 `127.0.0.1`；机器人通过出站长连接工作 |

## 5. 总体架构

```text
本地页面“扫描任务” ─┐
                     ├─> BitableTaskService ─> TaskCoordinator ─> 现有 LangGraph
飞书消息/卡片事件 ───┘          │                    │              │
                               │                    │              ├─ Claude Vision
                               │                    │              ├─ DeepSeek 规划/审查
                               │                    │              ├─ 人工审批中断
                               │                    │              ├─ Chiyun / Seedance
                               │                    │              └─ 产物校验
                               │                    │
                               └─ BitableResultWriter <────────────┘
                                          │
                                          └─ 上传媒体并更新“结果”附件列
```

### 5.1 `BitableTaskService`

职责：

- 解析配置的 `/wiki/<token>?table=...&view=...` 多维表格链接。
- 将 wiki 节点解析为 `obj_type=bitable` 和实际 `app_token`。
- 读取表格字段定义和指定视图的记录。
- 检查必需字段类型，创建允许自动补齐的字段。
- 判断记录是否可处理，构造不含飞书原始响应细节的领域对象。
- 协调领取、释放、状态、错误信息和结果冲突检查。

它不负责模型调用、生成执行或机器人展示，因此可以被本地 API、机器人和测试 Fake 独立复用。

### 5.2 `TaskCoordinator`

职责：

- 建立 `app_token + table_id + record_id → run_id → thread_id → claimant_open_id` 的稳定映射。
- 在 SQLite 事务内原子领取记录，保证同一记录最多有一个活动运行。
- 创建并恢复现有 LangGraph 工作流。
- 将内部状态映射到多维表格中文状态。
- 生成并验证 `approval_version`，使旧审批卡和重复点击失效。
- 为本地页面和机器人提供相同的任务视图与操作结果。

### 5.3 `BotGateway`

机器人使用飞书应用机器人和官方长连接能力。进程只主动连接飞书，不监听公网回调。

消息或卡片事件到达后，入口必须在 3 秒内完成以下动作：

1. 校验事件基本结构。
2. 以 `event_id` 或 `action_id` 写入持久化收件箱；重复 ID 直接返回已处理结果。
3. 对领取、审批等动作写入最小持久化意图。
4. 立即确认事件，并把读取文档、规划、生成、上传等耗时工作交给后台协调器。

机器人入口不得在事件处理线程内等待模型或供应商响应。进程启动时会扫描未完成收件箱，避免“已经确认事件但后台尚未开始”造成任务丢失。

### 5.4 `BitableResultWriter`

职责：

- 只接收已通过现有文件校验的本地产物。
- 使用飞书官方文件/素材上传接口取得可用于多维表格附件字段的文件 token。
- 在回写前重新读取记录，确认 `结果`仍为空且记录仍属于当前运行。
- 将一项或多项附件一次性更新到 `结果`字段。
- 持久化每个本地产物对应的上传 token，回写重试时复用上传结果，绝不重新生成。

当前 `FeishuDeliveryWriter` 继续服务旧兼容模式；表格模式通过新的结果写入端口替换图中的交付节点，不把表格逻辑塞进旧的新建文档实现。

## 6. 多维表格契约

### 6.1 配置

表格 URL、table ID 和 view ID 只通过本地环境配置提供，不硬编码到仓库。建议配置项：

```text
LARK_BITABLE_URL
LARK_BITABLE_TABLE_ID
LARK_BITABLE_VIEW_ID
LARK_LOCAL_OPERATOR_OPEN_ID
LARK_BOT_ENABLED
```

`LARK_BITABLE_URL` 包含 wiki token；若 URL 已带 `table` 和 `view` 查询参数，显式的 table/view 配置必须与 URL 一致，否则启动探针报错，不静默任选一个。

### 6.2 字段

| 字段 | 类型 | 用途 | 缺失时行为 |
|---|---|---|---|
| `文本` | 任意可显示文本/编号 | 列表展示标题，不作为主键 | 报准备度错误 |
| `需求来源` | 超链接或可解析 URL | 恰好一个飞书 docx/wiki 需求链接 | 报准备度错误 |
| `执行人` | 人员 | 领取成功后写实际领取人的 open_id | 报准备度错误 |
| `结果` | 附件 | 存放生成的图片和视频，可多选 | 报准备度错误 |
| `状态` | 单选 | 展示业务阶段 | 自动创建并补齐所需选项 |
| `错误信息` | 多行文本 | 最近一次可操作错误或冲突说明 | 自动创建 |

若四个既有业务字段存在但类型不兼容，系统不得尝试修改字段类型或迁移用户数据，只报告明确的准备度错误。自动创建动作只限 `状态`和 `错误信息`。

### 6.3 记录身份与可处理规则

记录的唯一业务身份是飞书 `record_id`，不是 `文本`中的序号，也不是需求文档 URL。

扫描结果只包含同时满足以下条件的记录：

- `需求来源`可以解析为恰好一个 docx/wiki 链接。
- `结果`为空。
- 飞书状态为空、`待处理`，或已有本地运行明确处于可安全重试的 `失败`终态。
- 本地不存在该记录的活动领取。

`处理中`、`待审批`、`生成中`、`回写中`、`回写失败`和`已完成`均不可创建新的生成运行。`回写失败`只能继续原运行的回写步骤。

扫描不因 `执行人`已有值而隐藏记录。按已确认的“所有人可领取”规则，最终成功领取者会覆盖执行人为自己的身份。

### 6.4 领取顺序

领取顺序固定为：

1. 在 SQLite 事务中为 `(app_token, table_id, record_id)` 建立唯一活动领取。
2. 创建 `run_id`、`thread_id` 和领取审计记录。
3. 将飞书 `执行人`更新为领取者、`状态`更新为 `处理中`并清空旧错误。
4. 启动文档读取和规划。

如果第 3 步失败，领取保留为“同步失败”并允许只重试飞书同步，不启动模型工作。不能先修改飞书再尝试本地占位，否则两个入口可能同时认为自己领取成功。

本地入口的领取人取自 `LARK_LOCAL_OPERATOR_OPEN_ID`；机器人入口取事件发送者的 open_id。两人并发领取时，SQLite 唯一约束只允许一人成功，另一人收到“已被领取”的当前状态和领取者信息。

## 7. 统一工作流与状态

### 7.1 表格模式数据流

```text
扫描可处理记录
  → 原子领取
  → 更新执行人/处理中
  → 读取需求文档和参考图片
  → Claude Vision 图片理解
  → DeepSeek 规划与独立审查
  → 待审批（LangGraph interrupt）
      ├─ 反馈重规划 → 新 approval_version → 待审批
      ├─ 取消 → 释放领取并回到待处理
      └─ 批准选中任务
  → 复查源文档版本
      ├─ 已变化 → 作废审批并重新规划
      └─ 未变化 → 生成中
  → 校验并下载产物
  → 回写中
  → 上传媒体并更新结果附件
  → 已完成
```

### 7.2 飞书状态

正常状态为：

```text
待处理 → 处理中 → 待审批 → 生成中 → 回写中 → 已完成
```

异常状态为：

- `失败`：读取需求、视觉理解、规划、审批后校验或生成失败。错误信息必须说明能否安全重试。
- `回写失败`：产物已经生成并通过校验，只允许重试上传或记录更新，不得重新调用 Chiyun/Seedance。

用户在生成前取消时，系统记录取消审计，清空 `执行人`，把表格状态恢复为 `待处理`，使任务可再次领取。取消不作为错误状态。

### 7.3 内外状态分离

飞书中文状态用于人类协作，不替代现有细粒度内部状态。供应商的 `intent_created`、`submitted`、`submission_uncertain`、`succeeded` 等状态继续保存在本地业务库；只有确认进入相应业务阶段后才映射到表格状态。

Graph State 不存储 API Key、App Secret、访问令牌、签名 URL、图片 Base64、完整飞书事件或完整供应商响应。

## 8. 机器人交互

### 8.1 扫描与领取

用户发送 `@机器人 扫描任务`（单聊可省略 `@`）后，机器人返回可处理记录卡片。卡片至少展示：

- `文本`展示值。
- 需求来源文档标题或链接摘要。
- 当前状态。
- “领取并分析”按钮。

列表分页或限制单次返回数量，避免超出卡片大小。领取按钮携带签名后的短期动作载荷，只包含记录定位信息和一次性 action ID，不包含密钥或完整需求内容。

### 8.2 审批卡片

规划完成后向领取者发送审批卡，展示：

- 源文档与版本摘要。
- 规划/审查结论和阻塞问题。
- 每个任务的类型、标题、完整提示词、参考图片用途、输出数量和视频参数。
- 可选择执行的任务。
- “批准选中任务”“反馈并重新规划”“取消任务”动作。

机器人首版不直接编辑任务字段。用户提交文字反馈后，协调器调用同一重规划路径，产生新的 `approval_version` 和新卡片。任何旧版本卡片的批准动作均返回“方案已更新，请使用最新卡片”，不能触发生成。

### 8.3 动作幂等与权限

- 每个消息事件按 `event_id` 去重。
- 每次卡片动作按 `action_id` 去重。
- 领取、批准、取消只允许成功一次；重复点击返回第一次的结果。
- 只有当前领取者可以批准、反馈或取消该运行。其他人仍可扫描其他任务，但不能操作已领取运行。
- 审批动作还必须匹配 `record_id + run_id + approval_version + claimant_open_id`。
- 批准只是恢复 LangGraph 中断；真正提交前仍执行现有 Pydantic 校验、源版本校验和提交意图幂等检查。

## 9. 并发、恢复和冲突处理

### 9.1 付费调用

现有 `run_id + task_id + operation` 提交意图仍是付费调用的第二层保护。若已经保存供应商官方 ID，恢复时只查询该任务；如果提交结果不确定，则进入现有 `submission_uncertain` 修复流程，不猜测成功、不自动二次提交。

### 9.2 进程重启

启动恢复规则：

- `处理中`：从安全节点继续读取、视觉分析或规划。
- `待审批`：恢复中断并重新发送当前版本审批卡，绝不自动批准。
- `生成中`：有官方 ID 则继续轮询；无安全提交证据时按现有不确定提交规则处理。
- `回写中`或`回写失败`：复用已校验本地产物和已有上传 token，只继续回写。
- 已确认完成或取消的运行不重新执行。

机器人长连接恢复后重新处理持久化收件箱中的未完成事件；飞书重发相同事件时由事件 ID 去重。

### 9.3 结果冲突

领取时要求 `结果`为空。生成完成准备回写时必须再次读取记录：

- 若 `结果`仍为空，继续上传和更新。
- 若发现人工或其他系统已经写入附件，停止回写，不覆盖、不合并，状态设为 `回写失败`，错误信息标记为“结果冲突，需要人工确认”。
- 若本地无对应运行但记录已有结果，扫描时直接排除。

飞书记录更新缺少跨系统事务，因此采用“本地唯一领取 + 回写前再读 + 不覆盖”的保守策略。它不能消除所有毫秒级外部竞态，但可以避免系统主动覆盖可见结果。

### 9.4 源文档变化

审批计划记录需求文档修订标识。批准后、首次付费调用前再次读取修订信息；若内容已变化，旧审批立即失效，状态回到 `处理中`并重新规划。只有新版本审批卡可以恢复生成。

## 10. 配置、模型与安全

### 10.1 能力化配置检查

配置探针按入口和能力分别检查，不再让旧交付文档所需的输出文件夹/固定协作者阻塞表格模式：

- 表格扫描：飞书应用身份、wiki/bitable 读取权限、字段契约。
- 表格领取/回写：记录写权限、人员字段写入、媒体上传和附件字段更新权限。
- 机器人：应用机器人、长连接、消息事件和卡片动作能力。
- 规划：Claude Vision 与 DeepSeek 模型可用性。
- 生成：Chiyun 模型与 Ark Seedance 模型可用性。
- 旧兼容交付：仅在使用旧入口时检查输出文件夹和文档协作者。

模型名必须通过供应商官方接口或实际无付费探针验证。用户期望使用 DeepSeek 4.0 Pro；若官方账户当前没有同名可调用模型，系统报告不可用并等待配置，不静默降级到较弱模型。Seedance 生成继续使用 Ark Bearer Key；另行提供的 AK/SK 和分组名不写入 `ARK_API_KEY`，除非未来增加明确需要 SigV4 的资产管理能力。

### 10.2 凭证与日志

- 所有真实凭证只存放在 gitignored 的 `.env`，权限保持 `0600`。
- 不把密钥写入设计文档、代码、测试夹具、数据库、Checkpoint 或日志。
- 日志只记录 provider 名称、模型名、官方任务 ID、哈希化幂等键和脱敏错误。
- 不持久化短期签名 URL；必须保存远端引用时只保存官方稳定 ID。
- LangSmith 默认关闭。

### 10.3 飞书权限

飞书企业应用需要按最小权限启用：

- 应用机器人与消息接收事件。
- 单聊消息和群聊中 `@机器人` 的事件权限。
- 多维表格应用、字段、视图和记录的读取/写入权限。
- wiki 节点与 docx 文档读取权限。
- 文档图片/媒体下载权限。
- 文件上传及多维表格附件写入所需权限。

固定表格、知识库和需求文档必须向该应用身份授权。权限准备由用户在飞书网页完成，应用只做探针和清晰报错，不尝试扩张企业权限。

## 11. 测试设计

### 11.1 自动化测试

新增测试至少覆盖：

1. wiki 多维表格 URL 解析为 bitable `app_token`，并校验 table/view 一致性。
2. 既有字段类型检查，以及只自动创建 `状态`/`错误信息`。
3. `需求来源`恰好一个 docx/wiki URL 的解析与拒绝规则。
4. 两名用户并发领取同一记录时只有一人成功。
5. 飞书事件和卡片动作重放不会重复领取、重规划或批准。
6. 反馈重规划会增加 `approval_version`，旧卡批准无效。
7. 重启后恢复待审批、生成中和回写中状态。
8. 多个图片/视频上传并一次写入附件字段。
9. 回写失败只重试上传/记录更新，不再次生成。
10. 发现既有结果附件时不覆盖。
11. 本地入口与机器人入口返回相同的记录和运行状态。
12. 现有后端与前端测试全部继续通过。

Fake 飞书客户端必须模拟分页、字段类型、并发记录变化、媒体上传、附件更新和事件重放。自动化测试禁止访问真实付费接口。

### 11.2 零生成联调

真实环境先运行不产生生成费用的探针：

- 飞书 tenant token 获取。
- wiki 节点解析为 bitable app token。
- 指定表格/视图的字段和记录读取。
- 人员字段、状态、错误信息和测试记录更新权限。
- 媒体上传与附件字段权限；使用无付费的小型测试文件，完成后清理测试记录内容。
- 机器人身份、消息事件、长连接和卡片动作接收。
- Claude、DeepSeek、Chiyun、Seedance 的 Key、模型可见性和请求权限。

随后用一条专用测试记录走通“扫描 → 领取 → 读取文档 → 规划 → 审批卡 → 取消”，确认付费生成调用计数为零。

### 11.3 付费冒烟

零生成联调通过后，使用专用测试记录并在飞书卡片明确批准：

- 生成 1 张最低合理规格图片。
- 生成 1 条最短、低分辨率、无音频视频。
- 确认两个产物都出现在该记录 `结果`附件列，状态为 `已完成`。
- 重启应用并再次恢复，确认没有新增供应商任务或重复扣费。
- 人为制造一次回写失败，确认修复后只重试回写，不重新生成。

任何真实付费冒烟都必须在运行当次再次获得用户明确同意。

## 12. 本地上线顺序与验收

### 12.1 上线顺序

1. 在隔离分支实现并跑完自动化测试。
2. 配置真实凭证和固定表格参数，运行只读/零生成探针。
3. 在飞书网页为应用开通机器人、事件和表格/文档权限。
4. 完成零生成端到端联调。
5. 经明确批准后完成一次最小付费冒烟。
6. 本机稳定运行后，再规划常驻主机迁移；迁移时只更换配置、进程管理和持久化目录，不重写核心工作流。

### 12.2 验收标准

- 本地页面和机器人扫描到同一批可处理记录。
- 任意有权使用机器人的成员都能领取可处理任务。
- 并发领取时只有一个领取者成功，`执行人`为该领取者。
- 本地页面可以完整编辑审批；飞书卡片可以选择任务、批准、取消和反馈重规划。
- 未审批时 Chiyun/Seedance 付费提交次数为零。
- 每个任务最多产生一次有效付费提交；重启或重复点击不增加提交。
- 图片和视频直接写入原记录 `结果`附件列，不新建交付文档。
- 正常流程按预期经过 `待处理/处理中/待审批/生成中/回写中/已完成`。
- 生成失败与回写失败可区分；回写重试绝不重新生成。
- 已有结果不被覆盖，源文档变化会使旧审批失效。
- 机器人在本机通过长连接工作，不需要公网 IP、域名或内网穿透。
- 所有真实密钥只存在本地安全配置中，测试和日志不泄露凭证。

## 13. 未来演进边界

为以后从本地 A 形态升级到常驻主机 C 形态，核心模块不得依赖固定 Mac 路径、固定 IP、launchd 或浏览器会话：

- `BitableTaskService`、`TaskCoordinator`、`BotGateway` 和 `BitableResultWriter` 使用明确端口/协议与依赖注入。
- 数据库和产物目录均由配置提供。
- 长连接生命周期与 FastAPI 生命周期解耦，可在未来拆为独立 worker。
- 持久化收件箱和幂等表允许未来多进程消费；首版本地进程仍只启动一个消费者。
- 未来若增加登录、队列、PostgreSQL、对象存储或公网管理页，不改变 `record_id → run_id → thread_id` 的业务身份和人工审批语义。

本规格只为这些演进保留接口边界，不在首版提前实现分布式队列、云存储或多租户系统。
