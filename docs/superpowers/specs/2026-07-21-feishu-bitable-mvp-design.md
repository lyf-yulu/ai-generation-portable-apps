# 飞书多维表格本地 MVP 设计

日期：2026-07-21

## 目标

在现有 `feishu-generation-agent` 上尽快打通一条可实际使用的本地链路：

1. 本地页面点击“扫描任务”，读取配置好的飞书多维表格。
2. 展示所有“需求来源有效且结果为空”的记录，不按“执行人”过滤。
3. 操作者手动选择一条记录并开始分析。
4. 复用现有飞书文档读取、Claude Vision、DeepSeek 规划和 LangGraph 审批流程。
5. 付费生成前必须在现有本地审批页面确认。
6. 复用现有 Chiyun 与 Seedance 执行能力。
7. 生成成功后把全部图片或视频作为附件写入原记录的“结果”列。

## MVP 范围

MVP 只使用四个现有字段：`文本`、`需求来源`、`执行人`、`结果`。

- `文本`：任务列表标题。
- `需求来源`：一个飞书 docx/wiki 链接。
- `执行人`：只展示，不过滤、不覆盖。
- `结果`：附件字段；为空才允许领取，成功后一次性写入全部产物。

MVP 不创建“状态”或“错误信息”字段。处理中、待审批、失败和冲突只在本地页面与 SQLite 中展示。失败或取消后释放本地领取，结果保持为空，可重新执行。

## 数据流

```text
本地页面扫描
  -> 飞书 Bitable 读取指定视图
  -> 过滤来源有效、结果为空、本地无活动领取的记录
  -> 操作者选择一条
  -> SQLite 原子领取
  -> 现有 GraphRuntime 读取/理解/规划
  -> 本地审批页人工批准
  -> Chiyun / Seedance
  -> 回写前重新读取记录并确认结果仍为空
  -> 上传产物并更新结果附件列
  -> 本地标记完成并释放领取
```

## 组件

### `FeishuBitableClient`

- 解析 wiki 节点得到 Bitable `app_token`。
- 校验四个既有字段存在且类型兼容，不自动修改用户表结构。
- 列出指定 view 的记录并转换为 `BitableTaskSummary`。
- 读取单条记录并写入结果附件。

### `BitableResultWriter`

- 实现现有 `DeliveryWriter` 协议。
- 回写前重新读取记录；结果非空时抛出冲突，不覆盖用户或其他进程的内容。
- 上传经过现有产物校验的本地文件。
- 持久化上传 token，使交付重试不重新生成、不重复上传。

### `BitableMvpService`

- 扫描可用记录。
- 用 `BitableTaskStore` 原子领取，领取身份固定为内部标识 `local-mvp`。
- 以预留的 `run_id/thread_id` 启动现有 `GraphRuntime`。
- 同步运行状态到本地绑定；失败、取消和完成时释放活动领取。
- 应用重启后恢复已有运行，不重复付费提交。

### 本地 Web 页面

- 增加“扫描多维表格任务”区域、刷新按钮和任务列表。
- 每条展示标题、需求来源和执行人；点击“开始分析”后进入现有运行详情。
- 审批仍使用现有完整审批界面，不复制第二套组件。
- 显示本地阶段、错误、结果冲突和可重试操作。

## 错误与安全规则

- 同一记录只能有一个活动本地领取。
- 飞书读取或领取前失败：不启动 Graph。
- 分析或生成失败：结果保持为空，释放领取，允许重新选择。
- 交付失败：保留已生成产物，只允许重试交付，不重新调用供应商。
- 结果列已被外部写入：停止回写并显示冲突，绝不覆盖。
- SQLite 不保存 API Key、原始飞书事件、图片 Base64 或带凭证的 URL。
- API 错误只返回可操作的中文摘要，不返回密钥或原始供应商响应。

## 验收

1. 使用 Fake 飞书与 Fake 生成器完成无付费端到端测试。
2. 使用真实表格完成只读扫描，能看到来源有效且结果为空的记录。
3. 使用专用测试记录完成零生成写回验证或上传小型测试附件后恢复记录。
4. 真实付费生成必须单独取得用户确认；在此之前只完成到审批门禁。
5. Python、前端和重启恢复测试全部通过。

## 延后到完整版

- 群聊或单聊 `@智能体`。
- 飞书长连接、消息去重和卡片动作。
- 任务列表卡、审批卡和结果卡。
- 多人领取者鉴权与写入“执行人”。
- 后台自动扫描和批量处理。
- 飞书表格中的“状态”和“错误信息”字段。

现有领域模型、原子领取存储和交付接口保留这些扩展点，MVP 不以临时代码绕开它们。
