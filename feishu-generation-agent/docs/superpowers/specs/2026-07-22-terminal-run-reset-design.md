# 终态任务复位与重跑设计

## 目标

让多维表格任务在生成、下载与飞书结果表写入完成后，网页明确收尾；使用者无需刷新即可继续开始下一条需求，也可以基于同一份已审批配置重跑。刷新页面后仍可查看最近完成的任务及其结果表链接。

## 范围与边界

- 适用当前多维表格生产流程；生产需求表始终只读。
- 终态为 `succeeded`、`completed_with_errors`、`failed`、`cancelled` 和 `delivery_failed`。
- 成功、部分成功、失败、取消都停止前端轮询；交付失败同样停止轮询，但保留既有“重试交付”入口。
- “开始下一任务”只清除浏览器当前运行上下文，随后重新读取可处理需求；不删除本地运行记录、不改写生产表、不触碰已经写入的结果表记录。
- “重跑此任务”仅允许从已经审批过且已经结束的多维表格运行启动。它复制最后审批通过的任务配置（提示词、尺寸/比例、时长、参考素材和参考模式），创建新的 `run_id` 与新的结果表记录。
- 重跑一定产生新的飞书结果表行；不覆盖原产物或原结果记录。用户可以在结果表中自行审核各版本。
- 不在本次功能中建设搜索、筛选或全量历史中心。
- 客户端只使用相对 API 路径和服务端返回的飞书链接；不保存本机路径、IP、密钥、SQLite 位置或本地运行 ID。服务迁移后客户端只需访问新的部署地址。
- 本地 SQLite 的建表与历史归档必须通过启动时幂等迁移完成；数据目录继续由现有服务端配置决定，不能写死 Mac 路径。

## 后端设计

### 最近任务列表

新增只读端点 `GET /api/bitable/recent-runs`，返回当前生产表绑定范围内最近完成的运行，按更新时间倒序，最多 10 条。每项只提供页面恢复所需字段：`run_id`、`display_text`、`status`、`updated_at`、`result_table_url` 与是否可重跑。

`ProductionTaskStore` 新增按 production location 查询终态绑定的接口；数据来源保持本地绑定库，不读取或修改生产需求表。现有 `active-runs` 语义保持不变，只用于恢复真正未结束的运行。

现有 `production_tasks` 以生产表 record 为主键，只能保存该需求的最新运行。为保留重跑前的本地详情，新增 `production_task_history` 本地表：当已释放的绑定被同一需求的新运行替换前，先将完整绑定复制到历史表。查询运行详情时同时查当前表与历史表；最近记录查询合并两表。该迁移仅创建本地 SQLite 表，不会删除任何数据，也不会写入生产需求表。

### 重跑

新增 `POST /api/bitable/runs/{run_id}/rerun`，返回新运行的 `run_id`。服务层执行以下检查：

1. 原运行必须有 production binding，且其状态已经释放为完成或失败；
2. 原运行必须存在可恢复的审批计划，且至少包含一个获批任务；
3. 从原运行的持久化审批状态创建新的独立运行，而不是恢复原图状态或复用原线程；
4. 新运行绑定相同的生产来源记录，以维持同一需求的互斥锁；旧运行已终态，锁不会阻止新版本领取；
5. 新运行进入正常审批后执行链路。为防止绕过用户确认，重跑会带着复制后的审批草案进入 `waiting_approval`，使用者可在页面上再确认或修改；只有再次点击批准才会调用付费生成。

每次重跑使用新的 `run_id`、`thread_id`、运行记录与结果表 record id，因此交付器自然创建新的结果表行。

## 前端设计

### 终态收尾

在 `app.js` 定义一个终态集合。`poll()` 收到终态后清除定时轮询器并渲染终态操作区：

- `succeeded`：显示“任务已完成”、结果表链接、“开始下一任务”和“重跑此任务”；
- `completed_with_errors`、`failed`、`cancelled`：显示对应终态与“开始下一任务”；如原运行满足重跑条件，显示“重跑此任务”；
- `delivery_failed`：保留“重试交付”，同时允许回到下一任务；不把交付重试误当作重新生成。

页面每次只维护一个可取消的轮询器，防止任务切换或完成后仍继续请求旧运行。

### 复位与最近记录

“开始下一任务”会清除 `runId`、`runMode`、审批草案、运行详情和事件区域，重新拉取任务扫描结果与最近终态列表。

页面初始化与每次任务收尾都会请求最近记录端点，在任务扫描区域显示最近 10 条简要记录。每条记录支持：

- “查看详情”：恢复该运行的只读详情和结果表链接；
- “重跑”：仅当后端标记 `rerunnable` 时显示，调用重跑端点并切换到新运行的审批页。

当前运行仍在进行时不展示“开始下一任务”，以免用户误解为取消或并行执行；取消仍使用既有取消按钮。

## 错误处理

- 最近记录读取失败只显示可恢复的提示，不影响扫描或正在进行的任务。
- 重跑端点返回 `409` 或 `422` 时，页面显示后端可理解错误，并刷新最近记录；不会创建半成品运行。
- 终态显示以后即使临时网络请求失败，页面保留最近一次成功取得的终态视图，不回退成“运行中”。

## 测试

- 后端单元/API 测试覆盖终态记录按时间倒序、只返回同一 production location、以及重跑创建新运行而不修改原绑定或原结果记录。
- 前端状态模块测试覆盖终态识别、复位清空和最近记录状态；DOM 测试覆盖终态后停止轮询、显示正确按钮与重跑请求。
- 集成测试覆盖：完成运行 -> 新结果表行 -> 重跑 -> 进入新的等待审批运行；不调用生成器直到第二次批准。
