# 生产表动画类路由与统一结果表设计

## 目标

恢复对新版生产多维表格的扫描，正式排除“已确认完成”任务；本期仅允许动画类任务进入生成流程。所有完成产物写入同一张结果表，不再按需求制作人创建独立结果表。

## 已验证的新版表结构

- 分类字段名称为“需求类型”，其中当前未完成记录均为“动画类”。
- “需求附件”已恢复为飞书超链接；21 条非“已确认完成”记录均可解析为可读取的飞书文档。
- 原扫描为空的原因是该列短暂变为纯文本标题，解析器因此跳过全部记录。
- 当前配置 `LARK_INCLUDE_COMPLETED_FOR_TEST=true`，导致扫描包含 23 条已确认完成记录；必须关闭。

## 扫描与领取规则

- 唯一状态排除规则：`当前进度 == 已确认完成` 的记录不读取、不显示、不领取；空进度也视为未完成。
- 扫描解析所有其他状态的记录，并保留 `需求类型` 到任务摘要与锁记录。
- 动画类：显示为可领取，沿用现有锁、审批、生成与完成释放逻辑。
- 真人类：显示为“暂未启用”，不得领取或发起运行。该分类不是错误，也不会影响动画类扫描。
- 未识别或为空的需求类型也显示为“暂未启用”，避免误调用动画生成链路。
- 需求附件无法解析为单一飞书文档 URL 时，仅跳过该条并在扫描响应中给出原因；不让一条坏记录清空整个列表。

## 配置

- `LARK_INCLUDE_COMPLETED_FOR_TEST=false` 为正式固定值；代码不再以该开关扩展已确认完成的可扫描集合。
- 动画类允许值使用集中常量 `动画类`，真人类和其他类别保留可扩展分类枚举/策略边界。

## 统一结果表

- 首次交付时，在已配置的飞书结果文件夹创建一张统一的“AI生成结果”多维表；之后所有任务复用它。
- 每项来源记录写入一行，字段顺序为：`需求名称`、`需求类型`、`需求附件`、`项目名称`、`发起人`、`需求制作人`、`结果`。
- “需求制作人”为空时保留空单元格，不阻止领取、审批或交付。
- 结果列仍保存飞书附件；重试交付更新同一行，不重复创建记录。
- 旧的按制作人结果表映射保留只读兼容，不删除既有表或历史结果；新运行一律使用统一表。

## 数据流

1. 扫描生产表，排除已确认完成，解析需求附件链接和需求类型。
2. 前端显示可领取动画类与只读真人类/未知类。
3. 用户领取动画类，服务端再次验证其类型，建立本地锁后读取需求文档。
4. 用户批准并完成生成；交付器将来源前六列快照与产物附件写入统一结果表。
5. 用户在结果表审核并自行更新生产表状态；应用不修改生产表状态。

## 错误处理与验收

- 无法解析附件、真人类领取、已确认完成领取，都返回明确中文原因。
- 一个附件无效或无制作人不会阻断其他记录扫描。
- 自动化测试覆盖：非完成状态筛选、动画类允许、真人类拒领、空制作人允许、统一表复用、字段复制与重试更新。
- 本地验证通过后，扫描应返回 21 条未完成记录（以当前生产表状态为准），其中动画类可领取；不再返回已确认完成记录。
