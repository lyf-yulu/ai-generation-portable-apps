# 生产需求表与按制作人结果表设计

## 目标

将飞书生产多维表格作为只读任务入口。用户在本地应用中选择需求、审阅并修改自动生成的任务拆解后，应用执行图像或视频生成。成功产物只写入按“需求制作人”分开的结果多维表格；生产表不新增、不修改任何字段或记录。

## 已确认的业务规则

- 生产表字段为：`需求名称`、`需求附件`、`项目名称`、`发起人`、`需求制作人`、`当前进度`、`优先级`、`父记录`。
- `需求附件` 必须是一个可读取的飞书 `docx` 或 `wiki` 链接；不满足时不显示为可选任务。
- 当前测试期展示 `未开始`、`制作中`、`待修改` 和 `已确认完成` 的可读需求。上线时可通过配置关闭 `已确认完成`。
- 每位需求制作人拥有一张结果多维表格。首次需要交付给该制作人时，应用在配置的结果文件夹内创建该表，并给该制作人编辑权限。
- 结果表一行对应一条生产需求。前五列复制生产记录的前五列值：`需求名称`、`需求附件`、`项目名称`、`发起人`、`需求制作人`；第六列 `结果` 保存生成产物附件。
- 制作人为空的需求可读取、可生成审批稿，但不得进入供应商提交或结果交付；界面必须提示先补齐生产表的制作人。
- 用户批准之前不得调用 Chiyun 或 Seedance，不得创建结果行。

## 方案选择

采用“自动按制作人建结果表”。不使用一张总表，也不要求预先人工创建每位制作人的表。

应用创建新的云空间多维表格，而不复制生产表。这样可在目标文件夹中创建干净的六列结果表，避免生产表中的历史记录、视图和权限被带入结果表。

## 数据模型与配置

新增独立的生产表配置，不复用旧 MVP 的 `文本 / 需求来源 / 执行人 / 结果` 固定字段配置：

- `LARK_PRODUCTION_BITABLE_URL`：生产表 wiki 链接。
- `LARK_PRODUCTION_TABLE_ID`、`LARK_PRODUCTION_VIEW_ID`：生产表表和视图标识。
- `LARK_RESULT_FOLDER_TOKEN`：结果表父文件夹标识。
- `LARK_INCLUDE_COMPLETED_FOR_TEST`：测试期开关，默认 `false`。

本地 SQLite 保存两类稳定映射：

1. `maker_result_tables`：以需求制作人的 `open_id` 为键，保存结果表 app token、默认 table id、URL、创建时间和最近验证时间。
2. `production_deliveries`：以生产表 app token、table id、record id 为联合唯一键，保存结果表记录 id、审批通过的运行 id 和交付状态。

这两个映射使重试、服务重启和重复点击批准都不会创建额外结果表或重复结果行。

## 运行流程

1. 扫描生产表，读取字段元数据并验证必需字段名称与类型。
2. 读取目标视图的记录，仅保留 `需求附件` 可解析且进度符合当前扫描策略的记录。扫描不会写生产表。
3. 用户在本地界面选择一条任务后，读取飞书文档、文本块和图片，形成任务拆解和审批稿。
4. 用户可修改任务、提示词、比例、时长和供应商参数。批准前停在审批门禁。
5. 批准时先校验制作人存在，再执行被批准的任务。
6. 所有产物校验完成后，查找或创建对应制作人的结果表；确保该制作人拥有编辑权限。
7. 以生产记录 id 幂等地创建或更新结果行，写入前五列快照和 `结果` 附件。
8. 本地运行详情显示结果表链接、结果行 id 和交付状态。

## 结果表结构

每张自动创建的结果表只包含一个默认数据表，字段顺序固定为：

| 顺序 | 字段 | 类型 | 来源 |
| --- | --- | --- | --- |
| 1 | 需求名称 | 文本（主字段） | 生产表 `需求名称` |
| 2 | 需求附件 | 文本 | 生产表 `需求附件` |
| 3 | 项目名称 | 多选 | 生产表 `项目名称` |
| 4 | 发起人 | 人员 | 生产表 `发起人` |
| 5 | 需求制作人 | 人员 | 生产表 `需求制作人` |
| 6 | 结果 | 附件 | 生成后的图片或视频文件 token |

创建成功后立刻为 `需求制作人.open_id` 添加 `edit` 协作者权限。若权限添加失败，交付标记为失败且可重试；不会改写生产表，也不会向其他位置降级写入。

## 飞书权限与失败策略

- 启动或首次交付前执行只读预检：生产表读权限、需求文档/图片读权限、结果文件夹访问权限、创建多维表格权限、创建字段/记录权限、添加协作者权限。
- 结果表必须创建在 `LARK_RESULT_FOLDER_TOKEN` 指定的文件夹中。任何文件夹权限不足、API scope 缺失或创建失败都中止交付，并显示可操作的错误说明。
- 不允许回退到应用根目录、其他文件夹或生产表写入。
- 交付成功后结果附件由结果表承载；本地文件继续保留在现有运行输出目录，供下载和重试使用。

## 界面与审批

- 任务列表显示生产表名称、项目、当前进度、制作人和文档链接；不显示无法读取附件的记录。
- 列表对制作人为空的记录显示“无法交付：缺少需求制作人”，但允许打开并阅读需求。
- 审批页保留现有任务选择与编辑行为；“批准”动作在制作人为空时禁用并给出原因。
- 交付完成页提供结果表链接和对应结果行状态。

## 测试与验收

- 单元测试：生产表字段映射、进度过滤、可读链接验证、缺失制作人门禁、结果表字段构造、人员权限请求、幂等映射。
- 集成测试：模拟飞书 API，覆盖首次建表、重复交付更新同一行、添加协作者失败、文件夹权限失败和服务重启恢复。
- 前端测试：生产任务展示、测试期完成状态开关、缺失制作人禁用批准、结果表链接显示。
- 实际接入顺序：先执行只读预检和生产表扫描；确认后只创建一张空的结果表进行结构与权限验证；最后启用真实交付。

## 非目标

- 不修改生产表的进度、结果或其他字段。
- 不为无制作人的任务创建“未分配”表。
- 不在审批前生成或创建结果行。
- 不迁移历史 859 条生产记录到结果表。
